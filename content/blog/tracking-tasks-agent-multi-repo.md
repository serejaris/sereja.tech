Если трек тянется больше двух недель и артефакты оседают в двух-трёх репо — это уже не issue. Это инициатива. Для неё в GitHub есть готовая форма: change set pattern. В феврале её описали William Salazar и Kitty Chiu в [GitHub Well-Architected](https://wellarchitected.github.com/library/architecture/recommendations/implementing-polyrepo-engineering/). Пять простых элементов делают трек читаемым и для тебя, и для AI-агента, который через месяц должен будет восстановить контекст.

## Момент, когда становится нужна форма

Вот сценарий, в который я упёрся. Партнёрский образовательный трек на два месяца. Деки — в одном репо. Программа и учебные материалы — в другом. Карточка сделки в CRM — в третьем. Плюс внешние ссылки: Luma, Zoom, чат с партнёром.

Я спрашиваю у агента: «что у нас открыто по этому треку?». Агент честно отвечает: не знает. Задачи размазаны по репо, в каждом свой backlog, связывающей нити между ними нет.

Раньше я писал [про GitHub Projects как память для AI-агента](/blog/github-projects-ai-agent-memory) — там речь шла про один репо: доска = память, агент читает и отвечает. Когда работа в одном месте, этого хватает. Но как только инициатива разрастается на несколько репо, одной доски недостаточно. Нужна форма, в которой доска, эпик, лейблы и коммиты собираются в одно целое.

## Пять элементов

Суть в том, что инициатива получает пять связанных «крепёжных точек». Ни одна из них по отдельности не революционна. Но вместе они дают агенту возможность ответить «что происходит» за пару секунд.

```
         ┌────────────────────────────────┐
         │         PROJECT                │  ← общая доска на
         │  (всё по инициативе здесь)     │     всю инициативу
         └────────────┬───────────────────┘
                      │
                      ▼
              ┌───────────────┐
              │     ЭПИК      │  ← один issue в «родном»
              │  (overview)   │     репо инициативы
              └───┬───────┬───┘
                  │       │
       ┌──────────┘       └──────────┐
       ▼                             ▼
 ┌───────────┐              ┌───────────────┐
 │ задача в  │              │  задача в     │
 │  репо А   │              │    репо B     │
 └───────────┘              └───────────────┘

 LABEL: общий маркер во всех репо ────────────┐
 COMMIT-ПРЕФИКС: [slug] в каждом коммите ─────┤ ← агент
 TRACKING: блок-указатель в README-ах ────────┘   найдёт
                                                   всё
```

**Project.** Общая доска на всю инициативу. Сюда стекаются задачи из всех репо, участвующих в треке. Не важно, где задача физически живёт — на доске она видна рядом с остальными.

**Эпик.** Один issue в роли overview. Живёт в «родном» репо инициативы: для образовательной программы — в репо программы, для B2B — в CRM. В теле эпика — что, зачем, на какой срок, ссылки. Все остальные задачи трека — дети этого эпика, даже если физически они в других репо. Агент открывает эпик — видит дерево.

**Label.** Общий маркер во всех репо, один и тот же короткий слаг. Резервный путь поиска: забыл номер эпика — фильтруешь по лейблу и получаешь все задачи трека кросс-репо.

**Префикс в коммитах.** Каждый коммит, который касается инициативы, начинается с `[slug]`. Агент читает историю и восстанавливает хронологию. Без догадок, без разбора 40 коммитов вручную.

**Блок Tracking.** В каждом артефакте (README, CRM-карточка, папка программы) — маленькая секция «Tracking» с тремя указателями: ссылка на Project, на эпик, имя label. Не копия содержимого эпика, а именно указатели. Открываешь папку через месяц — сразу видишь, куда идти за контекстом.

## Деталь, которая всё это скрепляет

Главная причина, почему это работает сейчас, а не год назад: **sub-issues эпика могут быть из другого репо**. GitHub сделал это официально — sub-issues стали GA в [апреле 2025](https://github.blog/changelog/2025-04-09-evolving-github-issues-and-projects/), а в [сентябре](https://github.blog/changelog/2025-09-11-a-rest-api-for-github-projects-sub-issues-improvements-and-more/) добавили REST API и улучшения для работы между репо. Раньше приходилось клеить задачи из разных репо через лейблы и надеяться, что ничего не развалится. Теперь открыл эпик — видишь все подзадачи, в каких бы репо они ни жили. В Project доступен Hierarchy view — агент читает дерево и отвечает «что открыто по этому треку» без дополнительных вопросов.

## Как это выглядит на живом треке

Сегодня я запустил такой трек — партнёрская образовательная программа на апрель-май. Три репо: деки, программа, CRM. Я не разбирался вручную, какие команды нужны, какие поля заводить, где что создавать. Протокол лежит в `~/.claude/rules/multi-repo-initiatives.md`. Попросил агента прочитать его и настроить инициативу.

Написал Claude Code на Opus 4.6:

{{< callout type="insight" >}}
Собери multi-repo initiative для инициативы `x26-bloom`: образовательный партнёрский трек на apr–may 2026, артефакты в teach-vibecoding, corp-decks, crm. Домен — образовательная программа. Разберись с протоколом (он у меня в `~/.claude/rules/multi-repo-initiatives.md`) и сделай всё, что там описано: Project, эпик, label в каждом репо, блоки Tracking в README-ах. Покажи мне итоговые ссылки.
{{< /callout >}}

Агент сам решил, где якорить эпик (программа — значит, в репо программы), сам создал Project, сам завёл label в трёх репо, сам добавил блоки Tracking в нужные README. Я получил три ссылки: доска, эпик, label. Дальше трек живёт сам: каждый коммит с префиксом, каждая новая задача привязывается к эпику.

## Через месяц

Когда через месяц понадобится вспомнить, где мы остановились — открываю чат с агентом:

{{< callout type="insight" >}}
Восстанови контекст инициативы `x26-bloom`. Покажи статус, что открыто, что закрыто, какие коммиты были за последний месяц. Мне нужно за 30 секунд вспомнить, где мы остановились.
{{< /callout >}}

Агент знает, куда смотреть: Project для текущего статуса, эпик для overview, label для истории issues, история коммитов с префиксом для хронологии. Собирает статус и отдаёт. Без раскопок. Форма сама подсказывает агенту, где контекст.

## Что нужно, чтобы это заработало

Под капотом агент использует `gh` CLI для работы с GitHub — создания Project, эпика, лейблов, tracking-блоков. Если у тебя `gh` не установлен — скажи агенту поставить его, он сам разберётся с brew / apt / установщиком под твою систему. Дальше агент попросит авторизоваться в GitHub через браузер — это одноразово, потом токен живёт в системе. И третье — сам протокол в файле `~/.claude/rules/multi-repo-initiatives.md`, чтобы агент читал его и знал, что делать. Попроси агента создать этот файл, если его ещё нет — он сам наполнит по описанию из статьи.

Всё. Дальше первый промпт из раздела выше запустит трек.

## Когда НЕ применять

Не на каждую задачу нужен такой протокол. Пять элементов — это инфраструктура, и она стоит смысла только когда трек действительно большой.

| Ситуация | Что вместо |
|----------|-----------|
| Одноразовая задача | Обычный issue в репо |
| Работа в одном репо | Issue + label, без Project |
| Текущий активный поток курса | Уже существующая доска потока |
| Встреча или созвон | Карточка в CRM |
| Research-исследование | Markdown-файл в research-репо |

Если задача делается за день — форма стоит дороже, чем приносит. Если работа идёт в одном репо — встроенной доски хватит. Протокол нужен только когда инициатива уже переросла одно место.

## Почему это работает

Change set pattern — это не моё изобретение и даже не community-лайфхак. Это официальная best practice GitHub, описанная в [Well-Architected](https://wellarchitected.github.com/library/architecture/recommendations/implementing-polyrepo-engineering/) 24 февраля 2026 года. Пять элементов дают агенту структуру, которую он читает без догадок. Проект — общий статус. Эпик — overview. Label — чем помечено. Префикс коммитов — хронология. Tracking-блоки в README — куда вернуться за контекстом.

Тот же принцип, что я описывал [в посте про синхронизацию Claude Code на четырёх компах](/blog/sync-claude-code-four-machines) — превращать инфраструктуру в source of truth, чтобы агент не догадывался, а читал. Там речь шла про конфиги. Здесь — про инициативы, которые растут больше чем на один репо.

Момент, когда инициатива перерастает один репо — это момент, когда ты даёшь треку форму. В ней трек останется читаемым через месяц, через три, через полгода. Агент — или ты сам — сможет вернуться и продолжить с того места, где остановился. Без раскопок. Без «а где мы вообще это обсуждали».
