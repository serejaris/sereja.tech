Claude Code через MCP или skill управляет NotebookLM — бесплатным RAG-движком Google на Gemini. Загружаешь записи занятий, PDF, ссылки — и получаешь ответы с цитатами из своих же источников. Два AI работают в паре: Gemini индексирует и ищет, Claude думает и действует. Я только пишу промпт.

## Проблема: RAG есть, API нет

NotebookLM — мощная штука. Загружаешь документы, Gemini индексирует, отвечает с точными цитатами. Бесплатно. Работает из браузера.

Но именно «из браузера» — это и проблема.

У меня 9 завершённых потоков курса. Записи занятий, транскрипты, конспекты. Я хочу спросить «в каком уроке я объяснял MCP?» — и получить цитату с номером минуты. NotebookLM это умеет. Но каждый раз открывать вкладку, создавать ноутбук, вручную загружать файлы, копировать ответ обратно в терминал — это не workflow. Это ручная работа.

Официальный API? Только для Enterprise. Preview, Pre-GA, корпоративная лицензия. Для обычного пользователя — заглушка.

Я уже [строил локальный RAG](/blog/local-rag-embeddings-m1/) на ChromaDB и эмбедингах. Работает, но требует настройки: модели, индексация, chunking. А тут Google уже сделал всё за меня — индексация, поиск, цитаты. Осталось подключиться.

## Решение: notebooklm-py

[notebooklm-py](https://github.com/teng-lin/notebooklm-py) — open-source обёртка вокруг undocumented API NotebookLM. 5.2K звёзд на GitHub, MIT-лицензия, версия 0.3.4. Автор reverse-engineered 77+ RPC-методов Google и завернул их в нормальный Python-клиент.

Ключевое: это не скрейпинг и не Selenium. Это прямые вызовы тех же RPC, что использует веб-интерфейс NotebookLM. Авторизация через Playwright — открывается Chromium, логинишься в Google, cookies сохраняются. Дальше всё работает без браузера.

## Архитектура: два AI в связке

```
┌──────────────────────────────────────────────────┐
│                  Claude Code                      │
│              (оркестратор, Opus 4.6)              │
│                                                   │
│  "создай ноутбук из записей потока #9,            │
│   найди где я объяснял MCP"                       │
└───────────────────┬──────────────────────────────┘
                    │ MCP / skill / CLI
                    ▼
┌──────────────────────────────────────────────────┐
│              notebooklm-py                        │
│         (Python, RPC-вызовы Google)               │
│                                                   │
│  notebooks.create() → sources.add() → chat()      │
└───────────────────┬──────────────────────────────┘
                    │ undocumented RPC
                    ▼
┌──────────────────────────────────────────────────┐
│              NotebookLM (Gemini)                  │
│         бесплатный RAG-движок Google              │
│                                                   │
│  индексация → семантический поиск → цитаты        │
└──────────────────────────────────────────────────┘
```

Суть: Claude Code не хранит мои записи в контексте. Он отправляет их в NotebookLM, где Gemini индексирует и ищет. А потом забирает ответ с цитатами и работает с ним дальше. Каждый AI делает то, в чём он лучше.

## Use case 1: research по записям занятий

Главное, ради чего я это делал. У меня накопились транскрипты 36 уроков за 9 потоков. Нужно быстро находить, где и что я объяснял.

{{< callout type="insight" >}}
**Промпт для Claude Code (Opus 4.6) с NotebookLM MCP:**

Создай ноутбук "Кружок Вайбкодинга — Записи". Добавь туда транскрипты из ~/Documents/GitHub/teach-vibecoding/. Потом спроси: в каких уроках я объяснял разницу между MCP и skill? Дай цитаты.
{{< /callout >}}

Агент создал ноутбук, загрузил транскрипты как источники, задал вопрос. Вернулся ответ с точными цитатами — номер урока, фрагмент текста, где именно я это говорил.

Раньше я искал grep-ом по файлам. Находил слово «MCP» в 15 местах и читал каждое. Сейчас — семантический поиск. Спрашиваю «где объяснял разницу», получаю три релевантных фрагмента с контекстом. Не keyword matching, а понимание смысла.

## Use case 2: подкаст из материалов

NotebookLM умеет генерировать подкасты — два голоса обсуждают загруженные материалы. Звучит как побочная фича, но для курса это полезно: студенты могут послушать пересказ урока по дороге на работу.

{{< callout type="insight" >}}
**Промпт для Claude Code (Opus 4.6):**

Возьми ноутбук с записями потока #9. Сгенерируй подкаст по материалам третьей недели — тема "первый деплой".
{{< /callout >}}

Агент запросил генерацию подкаста. NotebookLM выдал аудио — два голоса обсуждают, что такое деплой, зачем нужен Railway, как студенты справились. На основе моих же записей, не выдуманный контент.

Помимо подкастов, через тот же интерфейс можно генерировать квизы, слайды, инфографику, mind maps — всё из загруженных источников, одним вызовом.

## Use case 3: анализ URL с цитатами

Не обязательно загружать файлы. Можно скормить URL — и NotebookLM сам вытянет контент.

{{< callout type="insight" >}}
**Промпт для Claude Code (Opus 4.6):**

Создай ноутбук, добавь эти три статьи как источники: [URLs]. Сравни подходы авторов к prompt engineering. Цитируй каждый тезис.

Агент добавил URL, дождался индексации, задал вопрос. Ответ пришёл со ссылками на конкретные абзацы из каждой статьи. Не пересказ — цитаты.

Это, по сути, [data layer](/blog/data-layer-for-agents/) для исследований. Вместо того чтобы самому читать три статьи и сравнивать — загружаешь в NotebookLM, спрашиваешь у Claude Code, получаешь структурированный ответ.

## Под капотом: как работает авторизация

Для тех, кому интересна механика. Авторизация — самая хитрая часть.

Playwright открывает Chromium. Ты логинишься в Google — как обычно, через форму. Playwright перехватывает session cookies: SID, HSID, APISID, SAPISID. Сохраняет в `storage_state.json`.

При каждом запуске notebooklm-py делает GET на главную страницу NotebookLM, парсит из HTML два токена — CSRF и Session ID. С ними и cookies — полноценная сессия. Без OAuth, без API-ключа.

Когда Google меняет method IDs (а они это делают) — в notebooklm-py обновляется только один файл: `RPCMethod` enum. Архитектура четырёхслойная: CLI → Client API → Core → RPC Layer. Ломается только нижний слой, остальные стабильны.

## Два способа подключить к Claude Code

**MCP-сервер.** Есть готовый — [notebooklm-mcp](https://github.com/PleasePrompto/notebooklm-mcp). После подключения NotebookLM становится ещё одним инструментом агента рядом с Chrome DevTools, Exa и остальными.

**Skill.** Для большинства задач MCP-серверы избыточны, хватает CLI. notebooklm-py из коробки умеет всё — создавать ноутбуки, добавлять источники, задавать вопросы. Skill — обёртка, которая учит агента использовать этот CLI. Ставится одной командой.

Я использую skill. Меньше движущихся частей: не нужен отдельный процесс MCP-сервера, не нужно следить за его состоянием. Claude Code просто вызывает CLI-команды.

Это undocumented API. Google не обещал стабильность. Завтра могут сменить method IDs — и всё сломается до следующего обновления библиотеки.

Но. У проекта есть ночной мониторинг, который каждую ночь проверяет, живы ли методы Google. Если что-то ломается — автоматически создаётся issue. Поддержка активная: 5.2K звёзд, регулярные релизы.

Для production-систем — рискованно. Для research и подготовки уроков — более чем достаточно. Если API упадёт на день, я переживу.

Два AI лучше одного, когда каждый делает своё. Gemini в NotebookLM — индексирует документы и ищет с цитатами. Claude Code — думает, планирует, действует. notebooklm-py — мост между ними.

## FAQ

### NotebookLM правда бесплатный?

Да. Бесплатный для обычных пользователей Google. Есть NotebookLM Plus за $20/мес с увеличенными лимитами, но базовая версия покрывает мои задачи.

### Чем это отличается от локального RAG на ChromaDB?

Локальный RAG — ты сам управляешь эмбедингами, chunking, моделью поиска. Полный контроль, но и полная ответственность. NotebookLM — Google делает всё за тебя: индексация, поиск, цитаты. Ты только загружаешь источники. Компромисс: меньше контроля, но ноль настройки.

### Что если Google закроет undocumented API?

Официальный API для Enterprise уже существует — значит, Google не планирует убивать NotebookLM. Если undocumented API закроют, появится либо публичный API, либо community найдёт новые endpoints. Проект с 5.2K звёздами не умрёт тихо.

### Можно ли использовать для конфиденциальных данных?

Данные уходят в Google. Если это критично — лучше локальный RAG. Для учебных материалов, публичных статей и research — NotebookLM подходит.
