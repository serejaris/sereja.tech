Gemini 3.1 Pro генерирует SVG-анимации с параллаксом, слоями и переходами за один промпт. Без внешних библиотек — нативный SVG с `<animate>` и `<animateTransform>`. Google [выпустил модель](https://blog.google/technology/google-deepmind/gemini-3-1-pro/) 19 февраля 2026. За неделю я понял: узкий специалист. Но настолько хороший, что ему стоит держать место в арсенале.

## Почему я вообще полез в Gemini

На [стриме](https://youtube.com/live/8oJQyTai2X4) решил потестировать Gemini 3.1 Pro. Не потому что ищу замену Claude Code — а потому что слышал конкретное: эта модель умеет SVG. В Canvas-режиме результат видно сразу, не надо копировать код в файл и открывать в браузере. Написал промпт — увидел анимацию.

## Слабый агент, сильный генератор

Одна вещь, которую стоит сказать сразу: Gemini 3.1 Pro — плохой агент.

На Hacker News бывший инженер Google [написал прямо](https://news.ycombinator.com/item?id=47074735): "Stunningly good at reasoning and design, but falls over when trying to get things done. Gets stuck in loops. Bad at using tools." Блестяще рассуждает и проектирует, но спотыкается при исполнении. Застревает в циклах. Плохо использует инструменты.

Бенчмарки подтверждают. [SWE-Bench](https://www.swebench.com/) — Claude Opus 4.6 лидирует, Gemini заметно позади. Terminal-Bench — GPT-5.3 Codex. Визуальные задачи — SVG, дизайн, анимации? Тут Gemini вне конкуренции.

И тут возникает вопрос: зачем вообще держать модель, которая не умеет работать как агент? Ответ простой — потому что у неё есть одна вещь, которую другие модели не делают.

## Суперсилы по моделям

Каждая модель сильна в своём. Универсальных нет. Вот как я вижу расклад на февраль 2026:

```
┌──────────────────────────────────────────────────────┐
│              Суперсилы моделей (февраль 2026)        │
├──────────────────┬───────────────────────────────────┤
│ Задача           │ Лучшая модель                     │
├──────────────────┼───────────────────────────────────┤
│ SVG / анимации   │ ★ Gemini 3.1 Pro                 │
│ Агент / код      │ ★ Claude Opus 4.6  (SWE-Bench)   │
│ Терминал / CLI   │ ★ GPT-5.3 Codex  (Terminal-Bench) │
│ Рассуждения      │ ★ Gemini 3.1 Pro  (ARC-AGI 77%)  │
│ Batch / рутина   │   Kimi K2.5  (дёшево)            │
├──────────────────┼───────────────────────────────────┤
│                  │                                   │
│ Gemini:          │  plan, design, SVG, reasoning     │
│ Claude:          │  execute, refactor, agent work    │
│ GPT:             │  terminal, codex, shell tasks     │
│                  │                                   │
└──────────────────┴───────────────────────────────────┘
```

Gemini 3.1 Pro набрала 77.1% на [ARC-AGI-2](https://arcprize.org/) — больше чем вдвое по сравнению с Gemini 3 Pro. Рассуждает отлично. Исполняет — плохо. Архитектор с гениальными чертежами, который не умеет забить гвоздь.

На [Reddit r/singularity](https://www.reddit.com/r/singularity/comments/1r95ed7/) написали: "This might be the beginning of everyone's UIs starting to diverge." Если модель генерирует уникальные SVG-интерфейсы за один промпт — каждый продукт может выглядеть по-своему. Без дизайнера.

## Plan-in-Gemini, execute-in-Claude

На [Hacker News](https://news.ycombinator.com/item?id=47074735) появился термин, который точно описывает мой подход: "plan-in-Gemini, execute-in-Claude." Придумал в Gemini, исполнил в Claude. Раньше я пытался всё делать в одной модели — теперь вижу, что это как пытаться одним ножом и хлеб резать, и дерево строгать.

Gemini рисует SVG-анимацию для лендинга. Я смотрю результат в Canvas, прошу доработать — добавить слой, изменить скорость, поменять цвета. Когда результат устраивает, забираю SVG и передаю Claude Code для интеграции в проект.

Это ровно тот же принцип, что и с Agent Teams: каждый агент делает своё. Только здесь "команда" — это разные модели. Подробнее про координацию агентов писал в [Agent Teams в Claude Code](/blog/agent-teams-opus-4-6).

Похожий подход использовал для [делегирования дизайна рою агентов](/blog/agent-swarm-frontend) — там субагенты генерировали три варианта фронтенда параллельно. Gemini может стать ещё одним "сотрудником" в таком рое, со своей специализацией.

А если довести идею до конца — это уже [личная корпорация](/blog/personal-corporation-event-driven-agents), где каждая модель работает как отдел со своей зоной ответственности.

## Цена вопроса

Gemini 3.1 Pro стоит $2 за миллион входных токенов и $12 за миллион выходных. SVG-анимация — это обычно один-два промпта. Пара центов за результат, на который вручную уйдёт полдня.

Canvas-режим экономит отдельный шаг: не надо сохранять файл, открывать браузер, проверять. Результат виден мгновенно. Для итеративной работы — "сдвинь слой", "ускорь частицы", "добавь тень" — это критично.

## Зачем вайбкодеру несколько моделей

Вайбкодер не привязан к одному инструменту. Claude Code — основной рабочий агент. Но когда задача — сгенерировать SVG-анимацию для блога, лендинга, презентации — Gemini 3.1 Pro сделает это лучше.

Пока это лучшая модель для SVG-анимаций. Не лучшая модель вообще — а лучшая для конкретной задачи.

Держать одну модель на все случаи — как держать один нож на кухне. Можно. Но повар держит набор. Вайбкодер держит стек моделей, где каждая закрывает свою нишу.

## FAQ

### Gemini 3.1 Pro может заменить Claude Code?

Нет. Gemini 3.1 Pro плохо работает как агент — застревает в циклах, не умеет пользоваться инструментами. Для написания кода, рефакторинга и агентских задач Claude Opus 4.6 остаётся лидером. Gemini стоит использовать для визуальных задач: SVG, анимации, дизайн-прототипы.

### Нужен ли Canvas mode для работы с SVG?

Не обязательно, но он экономит время. В Canvas результат виден сразу — можно итерировать промптами без цикла "скопировать код → сохранить файл → открыть в браузере". Для SVG-анимаций это заметное ускорение.

### Какие SVG-анимации Gemini делает лучше всего?

Параллакс с несколькими слоями, анимированные иллюстрации (персонажи, объекты), интерактивные сцены. Всё — нативным SVG с `<animate>` и `<animateTransform>`, без JavaScript и внешних библиотек.
